Network Tokens for Merchant Integration Teams

This version is for engineers integrating Checkout.com as one processor in a broader payments stack. The key point: network tokenization is usually processor-managed by Checkout.com, but explicit provisioning APIs become relevant when you need deterministic control, migration coverage, or cross-platform token portability.

How to read these examples

In most Checkout.com integrations, your application stores a Checkout.com payment handle such as src_xxx or an instrument reference. The network token may exist underneath that handle without your application ever storing the raw token number. The examples below show where your code usually stops and where explicit token APIs start becoming useful.

Relevant API touchpoints: first card payment

If you want the first card payment to also request network token provisioning, the control point is the payment request itself. This is the normal starting point for explaining the concept to engineers.

Sample request

POST /payments { "source": { "type": "card", "number": "4242424242424242", "expiry_month": 12, "expiry_year": 2030 }, "amount": 6500, "currency": "USD", "processing_channel_id": "pc_123", "processing": { "provision_network_token": true } }

Your merchant code still stores and reuses a Vault instrument or card source reference. The difference is what Checkout.com chooses to present downstream to the schemes and issuer during authorization.

  • Your app submits FPAN once, but asks Checkout.com to provision a network token in parallel.
  • In asynchronous mode, the initial authorization may still run on FPAN while tokenization completes in the background.
  • Later payments can then benefit from tokenized rails without changing your basic application contract.
  • Card lifecycle updates are maintained by the network, not by the cardholder re-entering details.

    • Stored card example: later merchant-initiated payment

    This is the example most merchants actually operate in production. Your application stores the Checkout.com handle and reuses it later. Checkout.com may use the network token underneath without asking your application to submit token details.

    Sample stored card charge

    POST /payments { "source": { "type": "id", "id": "src_wmlfc3zyhqzehihu7giusaaawu" }, "amount": 1500, "currency": "EUR", "processing_channel_id": "pc_123", "stored_credential": { "type": "recurring", "initiator": "merchant" } }

    For a standard Checkout.com Vault flow, your first implementation goal is still card capture, consent, storage, and later reuse of the instrument identifier. In many cases, no separate network-token code path is needed.

  • This is the merchant-facing stored-card contract: your code keeps src_xxx, not the PAN and usually not the network token number.
  • If Checkout.com has a network token available, it can optimize this payment path behind the scenes.
  • This is why network tokens and merchant-stored cards are complementary rather than competing designs.
  • Observability shifts to provisioning success metrics and token lifecycle events.

    • Explicit provisioning for an existing stored card

    This is the API you teach when the merchant already has Vaulted cards and wants to backfill tokenization before the next billing cycle, instead of waiting for the next live payment to trigger it.

    Sample explicit provisioning request

    POST /network-tokens { "source": { "type": "id", "id": "src_wmlfc3zyhqzehihu7giusaaawu" } }

    When explicit provisioning matters

  • This is useful for backfill campaigns across an existing stored-card portfolio.
  • A 200-style outcome means a token already existed; a 201-style outcome means Checkout.com provisioned one now.
  • This is the moment where explicit token APIs add value beyond the normal stored-card flow.
  • Exporting network tokens and cryptograms for another processor or acquirer.

    • Exporting the token and requesting a cryptogram

    Only teach this section when the merchant needs network-token portability across processors or acquirers. Many teams never need this in a standard single-processor Checkout.com deployment.

    Sample portability flow

    GET /network-tokens/nt_xgu3isllqfyu7ktpk5z2yxbwna POST /network-tokens/nt_xgu3isllqfyu7ktpk5z2yxbwna/cryptograms { "transaction_type": "recurring" }

    When explicit provisioning does not matter

  • This is the advanced path for multi-processor portability, not the default path for most merchants.
  • The network token number alone is not enough; the cryptogram is part of the transaction authorization package.
  • If the merchant never needs token portability, this whole section can usually stay out of their application code.
  • You are not migrating a pre-existing portfolio of stored cards.

    • What your database usually stores

    For most merchants, the correct stored-card implementation is not to store PAN or even the network token number. It is to store Checkout.com references plus business metadata and consent state.

    Sample merchant-side record

    { "customer_email": "alice@example.com", "checkout_customer_id": "cus_abc123", "checkout_instrument_id": "src_wmlfc3zyhqzehihu7giusaaawu", "scheme": "visa", "last4": "4242", "consent_type": "recurring", "processor": "checkout_com" }
    • This record is enough for normal stored-card reuse with Checkout.com.
    • It separates merchant-side state from sensitive card credentials.
    • In many implementations, the raw network token never needs to enter this table at all.

    How to explain the difference to your team

    Use this framing in architecture review meetings: the stored card handle is the object your application integrates with; the network token is the scheme-grade credential Checkout.com may use underneath to improve auth performance and lifecycle resilience.

    • Merchant-stored card answers: what does my app save and reuse?
    • Network token answers: what optimized credential reaches the schemes and issuer?
    • Explicit provisioning answers: when do I need direct operational control over token readiness?
    POST /network-tokens/{id}/cryptograms
    Question If the answer is yes Implication
    Do you already store cards with Checkout.com Vault? Automatic provisioning often covers the normal path. You may not need additional application logic.
    Do you have a portfolio of old stored cards needing tokenization now? Use explicit provisioning or a provisioning campaign. You avoid waiting for each card to be tokenized only when next used.
    Do you need token portability across processors? Retrieve network token details and request cryptograms. You are moving from processor-managed convenience to explicit token operations.
    Do you require first-transaction token use instead of FPAN fallback? Evaluate synchronous provisioning with Checkout.com support. Operational control increases, but latency and dependency on provisioning also increase.
    Do you need failure analytics for provisioning? Monitor provisioning results and lifecycle updates. Token observability becomes part of your payments operations model.

    Critical distinction to explain to engineering teams

    A Checkout.com Vault instrument is your merchant-facing reusable handle. A network token is the scheme-facing credential that can sit underneath it. In many integrations, the first is what your application stores and the second is what the processor optimizes with behind the scenes.

    Implementation checklist for merchants using Checkout.com as one processor

    • Confirm network tokenization is enabled on the relevant processing channel.
    • Ensure stored credential consent and MIT/CIT indicators are implemented correctly.
    • Track whether payments were processed using FPAN or tokenized rails when available.
    • Decide whether legacy portfolio token backfill is needed.
    • Decide whether you need token export and cryptogram retrieval for multi-processor use.
    • Subscribe to lifecycle notifications if token state changes matter to your platform operations.
