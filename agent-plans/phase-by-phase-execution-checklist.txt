7) Phase 3 - Runtime State Engine
============================================================
Primary owner
- Frontend State Logic Agent

Required documents to consume
- integration-lifecycle-demo-plan.txt
- component-id-registry.txt
- scenario-content-schema-spec.txt
- startup-page-field-spec.txt

Execution checklist
- Implement state shape exactly as defined in the plan/schema contracts.
- Treat startupContext as required parent state.
- Bind startup context into the demo runtime.
- Implement scenario/step/tab/language/audience/vertical transitions.
- On demo initialization, derive state.currentScenarioId from startupContext.paymentStack using the scenarioMapping defined in startupBindings.
- Expose enabledPaymentMethods from startupContext to UX stage renderer for payment widget visibility.
- Expose threeDsEnabled from startupContext to step renderer for 3DS step show/hide logic.
- Expose storedCardStrategy and paymentType to content renderer for conditional content branching.
- Implement BACK behavior.
- Implement EXIT behavior returning to startup page.
- Ensure language changes do not reset scenario/step unexpectedly.
- Ensure vertical changes do not leak stale content.
- Expose state snapshot to PDF export system (read access to startupContext and all step data).
- Add state.isExportingPdf flag to prevent navigation during export.

Outputs
- integration-lifecycle-studio.js state manager
- startup-to-demo state persistence logic

Handoff gate
- Runtime state keys match approved contract.
- BACK and EXIT are deterministic in all known paths.
- paymentStack correctly initializes scenario on demo entry.
- PDF export system can read state without triggering state transitions.
