Reserved Component ID Registry
Source Plan Version: 1.5
Priority: Create this before scenario rendering work so multiple agents can build safely in parallel.
Purpose: Define stable DOM/component identifiers and ownership boundaries for the integration lifecycle demo.

============================================================
1) Why This Document Exists
============================================================
This registry is the first coordination contract for parallel implementation.

It prevents:
- duplicate or conflicting DOM IDs
- UI/state wiring mismatches
- brittle test selectors
- merge conflicts between frontend, state, testing, and oversight workstreams

This document should be treated as authoritative for reserved IDs.
No agent should rename or repurpose these IDs without Oversight approval.

============================================================
2) Parallel-Safe Implementation Order
============================================================
Recommended order
1. Startup page field spec
2. Reserved component ID registry
3. Scenario-content schema spec
4. UI shell implementation
5. State wiring and rendering

Reason
- Agents can build markup, bindings, and tests safely only when IDs are stable.

============================================================
3) Global Naming Rules
============================================================
Rules
- Use kebab-case only.
- IDs must be unique across the entire demo experience.
- Prefix startup-page elements with startup-.
- Prefix demo runtime elements with demo-.
- Prefix scenario-specific repeated containers with data attributes, not duplicate IDs.
- Repeated items must use data-role/data-step/data-scenario attributes instead of repeated IDs.

============================================================
11) Repeated Element Policy
============================================================
Repeated collections must not create unique IDs for every item unless explicitly reserved here.
Use data attributes instead.

Required data attributes
- data-scenario-id
- data-step-index
- data-tab-id
- data-vertical
- data-audience-mode
- data-line-group
- data-webhook-event

Suggested repeated roles
- data-role="timeline-step"
- data-role="ux-page"
- data-role="code-block"
- data-role="webhook-event"
- data-role="config-row"

============================================================
12) Ownership Boundaries
============================================================
Frontend Experience Agent
- Owns structural markup IDs and layout wrappers.

State Logic Agent
- Owns event binding to reserved controls and navigation/state transitions.
- Must not rename IDs.

Developer Education Agent
- Owns rendering inside code/config/webhook panels.
- Must reuse demo-code-panel, demo-webhook-panel, and tab panel IDs.

Test and Integration Agent
- Uses these IDs as stable selectors.
- Must report any missing or duplicated ID as a contract violation.

Oversight Agent
- Arbitrates conflicts and approves changes.

============================================================
13) Change Control
============================================================
Any requested change to a reserved ID requires:
1. documented reason
2. Oversight approval
3. update to this registry before code changes are merged
4. update to test selectors and implementation contract references

============================================================
14) Minimum Acceptance For UI Contract Compliance
============================================================
- All required reserved IDs exist exactly once where applicable.
- No duplicate reserved IDs appear in DOM.
- Repeated structures use data attributes instead of copied IDs.
- BACK and EXIT controls are present and bindable.
- Startup gate inputs use the exact startup-* IDs defined here.
- Technical console tabs and panels use the exact demo-tab-* and demo-panel-* IDs defined here.
