Integration Lifecycle Demo Plan
Version: 1.5
Audience: Merchant commercial leaders + merchant integration engineers + solution architects
Goal: Build a new page (not a fix of existing auditorium pages) that clearly teaches end-to-end checkout lifecycle with side-by-side UX + frontend code + backend logic + webhook timeline.

============================================================
1) Mission And Non-Negotiables
============================================================
Primary outcome
- Create a dedicated demonstration experience for merchant integration teams that explains payment lifecycle from user journey to backend orchestration and webhook-driven state transitions.

Hard requirements from stakeholder
- Add a startup page before the presentation experience to configure demo context.
- Startup page must collect and apply:
  - Merchant vertical
  - Website locale
  - Order-item currency
  - Merchant name
  - Website text content shown during demo
- Startup page must define and expose the exact configurable website text content set required by the demo.
- Configurable website text must align to the chosen merchant vertical and reference the display pattern of:
  - gift-cards journey for digital-goods
  - phone-case-catalog journey for physical-goods
- Startup page is mandatory-gated: user must complete required setup fields before entering the demo website.
- Build a NEW page for presentation and technical walkthrough.
- Show side-by-side:
  1. Web UX behavior
  2. Frontend implementation details
  3. Backend request/response logic
  4. Webhook events at each lifecycle moment
- Add a merchant vertical switch:
  - Digital-goods mode (example: gift cards journey)
  - Physical-goods mode (example: phone case catalog journey)
  - UX must clearly differ by vertical, including shipping address requirement in physical-goods flow and no shipping address in digital-goods flow.
  - Digital-goods showcase must explicitly reference and mirror the express checkout journey style used in the gift-cards page.
- Cover multiple frontend integration approaches:
  - Checkout Flow component
  - HPP redirect
  - Custom-built UI with Payments API
- Include detailed configuration examples (parameters passed vs omitted and resulting behavior).
- Include lifecycle moments at least for:
  - User authorization
  - Capture success
  - Payment decline path
- Keep content understandable for both commercial and engineering audiences.
- Webhook display rule:
  - Approved/declined outcomes are to be driven by real test-card execution during demo.
  - Presentation should display webhook/event output returned by the actual payment result path.
  - Fallback rule: if a real webhook receiver cannot be implemented in this phase, use pre-scripted branch simulation to show webhook events for approved/declined/captured branches.
- Desktop-only presentation target:
  - The experience is designed for desktop width only.
  - Use full available desktop screen width.
  - Mobile compatibility is not required for this project.
- In-demo navigation controls:
  - Every demo page must have a BACK button for previous-page navigation.
  - Every demo page must have an EXIT control that returns directly to the startup page.
- Language scope rule:
  - UX/explanatory presentation supports English + Simplified Chinese.
  - Code and webhook payload display remain in English only.
- Code panel visual requirement:
  - Side-by-side code display must use a black coding-software-style background to clearly distinguish code from website text content.
- PDF export requirement:
  - The user must be able to export the entire presentation as a downloadable PDF file.
  - PDF content must include:
    - A configuration cover section: startup page settings (merchant name, payment stack, vertical, currency, enabled payment methods, 3DS status)
    - Per-step sections for the chosen scenario: UX rendering, code panel content, webhook events
  - PDF export must be accessible from a dedicated export control visible during the demo.
  - PDF output must be self-contained and readable without live browser interaction.
  - PDF generation must use a browser-compatible approach with no server-side dependency (for example, html2pdf.js via CDN).
  - Exported UX text must reflect startup context substitution (merchant name, product content, etc.).
- Startup page payment configuration requirement:
  - The startup page must configure the payment stack and behavioral parameters of the demo, in addition to merchant branding context.
  - Payment stack selection: which integration type to showcase (Flow component, HPP redirect, Full card payments API). This selection pre-determines the primary scenario shown in the demo.
  - Stored card strategy: Remember Me / self-built stored card / no aids for returning buyer. This selection determines whether and how returning-buyer stored-card capabilities are highlighted.
  - Payment type: one-time payment or recurring payment. Drives emphasis on recurring credential configuration in technical content.
  - Payment method selection: checkboxes for card, Apple Pay, Google Pay (default: all three enabled). Determines which payment method option buttons/widgets appear in the UX stage.
  - 3DS enabled toggle: controls whether the 3DS authentication step is shown in relevant flows.
  - These payment configuration selections must drive the scenario shown, the payment method widgets rendered in the UX stage, and the configuration parameters highlighted in the technical console.
- Product content prefill requirement:
  - Product name, product description, and product price must be prefilled by the Product Writer Agent with commercially realistic placeholder values.
  - These fields must remain fully editable by the user on the startup page before entering the demo.
  - Product price display must be consistent with the selected currency.

Out of scope for this project
- Refactoring existing non-technical and technical auditorium pages.
- Full production hardening or complete PCI architecture implementation.
- Live external dependency on third-party docs at runtime.

3) Experience Blueprint (What The Audience Sees)
============================================================
Top-level layout
- Header: Page purpose, audience mode selector, merchant vertical switch, scenario selector, BACK, EXIT.
- Main split view:
  - Left panel: "UX Stage" (visual checkout behavior playback)
  - Right panel: "Technical Console" (tabs for Frontend, Backend, Config, Webhooks)
- Bottom timeline rail:
  - Step-by-step lifecycle nodes with status states (pending, current, success, failed)

Recommended tabs in Technical Console
- Frontend tab:
  - Rendered code snippet for current scenario + current lifecycle step
  - Highlights active lines relevant to the step
- Backend tab:
  - API route and payload mapping for that step
- Config tab:
  - "Pass this parameter" vs "Omit this parameter" examples
  - Behavior delta explanation
- Webhooks tab:
  - Simulated inbound event JSON
  - Event order and idempotency hints

Code panel visual treatment
- Code areas in Frontend, Backend, and Webhooks tabs must use black background and coding-editor-like appearance.
- Ensure sufficient contrast for projector usage.

Audience mode switch
- Commercial mode:
  - Simplified language
  - Focus on UX impact, conversion, trust, retry handling
- Engineering mode:
  - Full payload details, event schemas, implementation gotchas

Merchant vertical switch
- Digital-goods mode:
  - No shipping address collection
  - UX must mirror gift-cards express checkout showcase behavior
- Physical-goods mode:
  - Shipping address collection visible and required in relevant steps
  - UX resembles phone-case catalog checkout path
- The same technical console should update contextually for each vertical where behavior differs.

Language support
- English + Simplified Chinese for UX labels/explanations only
- Code snippets and webhook payloads remain English-only

Viewport support
- Desktop-only target; optimize for projector and wide monitors
- Use full-screen width with generous split-pane real estate
- Do not allocate implementation effort to mobile adaptation in this phase

Startup flow and navigation behavior
- Step 1: User lands on startup setup page.
- Step 2: User completes all required setup fields.
- Step 3: User enters demo website with setup context applied.
- During demo: BACK navigates to previous demo page/step; EXIT always returns to startup page and resets or rehydrates setup according to chosen state policy.

Startup text-content model
- The startup page must define which website texts are configurable for presentation.
- Required configurable text groups:
  - Global brand and checkout text
  - Product-listing text
  - Product detail/supporting text (including product name, description, and price — prefilled by Product Writer)
  - Cart / order summary text
  - Success / confirmation text
- Digital-goods text model must reference gift-cards journey patterns, including item-card display, category/title text, short benefit text, CTA language, and delivery-oriented confirmation text.
- Physical-goods text model must reference phone-case-catalog journey patterns, including catalog headers, item names, item short descriptions, product detail/support copy, shipping-related text, and cart summary language.
- The plan must treat these texts as merchant-configurable demo inputs rather than hardcoded copy.

Startup payment configuration model
- The startup page must configure the payment integration path and checkout behavior for the demo:
  - Payment stack: selects which integration scenario is demoed (Flow / HPP / Full card payments API)
  - Stored card strategy: configures returning-buyer experience shown in the demo
  - Payment type: one-time vs recurring — drives recurring credential examples in technical console
  - Payment methods: checkbox multi-select determining which checkout payment options are rendered
  - 3DS: toggles whether strong authentication step is shown in the authorization flow

4) Scenario Matrix (Core Teaching Paths)
============================================================
Scenario A: Flow component card payment
- Frontend: Load Flow component, session token handling, callbacks
- Backend: Create payment session endpoint
- Config examples:
  - enabled_payment_methods / disabled_payment_methods
  - success_url / failure_url / cancel_url
- Webhooks:
  - payment_approved OR payment_declined
  - payment_captured (if capture happens)

Scenario B: HPP redirect
- Frontend: Create payment link request and redirect
- Backend: Hosted payments request body construction
- Config examples:
  - processing channel, customer object, return links
  - capture mode choices
- Webhooks:
  - payment_authorized
  - payment_captured / payment_declined

Scenario C: Custom card UI + Payments API
- Frontend: Secure card collection orchestration pattern (demo-safe abstraction)
- Backend: Direct /payments call pattern
- Config examples:
  - stored_credential flags (CIT/MIT style context)
  - network token provisioning flag behavior
- Webhooks:
  - authorization result
  - delayed capture and decline branch

Scenario D: Stored card follow-up payment
- Frontend: Returning user selection flow
- Backend: source.id/instrument reuse path
- Config examples:
  - recurring flag strategy
  - when to request explicit provisioning vs rely on automatic behavior
- Webhooks:
  - recurring payment events and failure handling

Scenario E: Vertical-differentiated checkout journey
- Digital-goods branch:
  - Gift-card express checkout style flow with email delivery, no shipping step
  - Showcase baseline references existing gift-cards express checkout behavior
- Physical-goods branch:
  - Catalog/cart style flow with shipping form and fulfillment-focused fields
- Technical differences to teach:
  - Validation and payload differences for address-related fields
  - UX and backend behavior delta driven by vertical switch

6) Parallelization Plan
============================================================
Parallel Track A
- Phase 1 (content model)
- Phase 2 (layout shell)
- Phase 2.5 (startup page)
Rationale
- Fastest path to visible progress while keeping clear contract boundaries.

Parallel Track B
- Phase 5 (scenario consistency review)
- Phase 6 (accessibility/presenter controls)
- Phase 6.5 (product writing and language review)
Rationale
- One track hardens correctness, one track hardens usability.

Phase 2.8 (PDF export system)
- Shell and export trigger can start in parallel with Phase 3.
- Full content capture and binding requires Phase 3 and Phase 4 to complete first.
Rationale
- Prevents PDF system from blocking the critical state engine and rendering engine paths.

Sequential constraints
- Phase 3 must wait for stable schema from Phase 1 and shell from Phase 2.
- Phase 3 must also wait for startup contract from Phase 2.5.
- Phase 2.8 full implementation requires Phase 3 and Phase 4.
- Phase 7 requires all tracks merged, Oversight signoff from Phase 5.5, Phase 2.8 stable, and placeholder/wording review from Phase 6.5.

8) Acceptance Criteria
============================================================
Functional
- At least 5 complete scenarios available, including vertical-differentiated journey.
- Each scenario has synchronized UX, frontend code, backend code, config notes, and webhook events.
- Startup page exists and blocks demo entry until all required setup fields are complete.
- Startup page includes payment configuration section: payment stack, stored card strategy, payment type, payment methods, and 3DS toggle — all required before demo entry.
- Startup page product content fields (product name, description, price) are prefilled by Product Writer but remain editable by the user.
- Startup page explicitly defines the full configurable website-text set and exposes the required subset as mandatory inputs.
- Language toggle supports English and Simplified Chinese for UX/explanatory text.
- Code and webhook displays remain English-only.
- Merchant vertical switch is available and clearly changes digital-goods vs physical-goods UX behavior.
- Digital-goods vertical demonstrates gift-cards-style express checkout behavior with no shipping collection.

9) Risks And Mitigations
============================================================
Risk: Content overload for technical audience
Mitigation
- Progressive disclosure with tabbed right pane and step-driven reveals.

Risk: Visual crowding in split layout
Mitigation
- Cap line length, increase whitespace, and allocate wider desktop pane ratios for dense technical content.

Risk: Desktop-only scope accidentally diluted by mobile-first compromises
Mitigation
- Explicitly optimize for desktop widths and defer mobile treatment to a later phase.

Risk: Vertical switch does not produce meaningful UX differences
Mitigation
- Define mandatory visible deltas (shipping form presence, validation fields, and payload differences).

Risk: Startup setup is bypassed, causing inconsistent demo context
Mitigation
- Enforce required startup gate and block entry until validation passes.

Risk: BACK and EXIT navigation logic diverges between pages
Mitigation
- Define a strict navigation contract and validate through deterministic route tests.

Risk: Agent work collisions across schema and component naming
Mitigation
- Oversight Agent enforces implementation contract and blocks non-compliant merges.

Risk: Webhook narrative drifts from actual run result
Mitigation
- Bind webhook panel to real returned result path from executed test-card scenario in demo runs.

Risk: Real webhook receiver not ready within phase timeline
Mitigation
- Provide explicit fallback to pre-scripted branch simulation for approved, declined, and captured event paths.

Risk: Inconsistent bilingual updates
Mitigation
- Single key-value content model; no duplicated inline text outside model.

Risk: Startup text fields are under-scoped and fail to cover visible merchant-facing content
Mitigation
- Define full website-text inventory up front and map each input to a specific visible UI area.

Risk: Placeholder copy sounds unnatural in real merchant presentations
Mitigation
- Product Writer Agent performs late-stage commercial wording review and placeholder curation.

Risk: Webhook confusion due to async behavior
Mitigation
- Add explicit "possible order variance" notes and retry/idempotency hints per event section.

Risk: PDF output is incomplete or fails to render all steps
Mitigation
- Phase 2.8 must validate PDF generation across all 5 scenarios before Phase 7 QA.
- Fallback: if html2pdf.js fails, provide window.print + print-specific CSS as a secondary path.

Risk: Payment configuration on startup page does not visibly affect demo content
Mitigation
- Content Modeling Agent must define content branches for: paymentStack, enabledPaymentMethods, threeDsEnabled, storedCardStrategy, paymentType.
- Test and Integration Agent must verify each configuration switch produces observable demo differences.
